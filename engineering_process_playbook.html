Product Planning & Roadmap
Quy trình từ ý tưởng → PRD → backlog → sprint. Đảm bảo mọi thứ được build đều có lý do rõ ràng.
💡
Nguyên tắc cốt lõi: Không có ticket nào được vào sprint nếu chưa qua Definition of Ready. Không có feature nào được build nếu chưa gắn với OKR hoặc metric rõ ràng.
01
Quarterly Roadmap Planning
Mỗi quý · 2 ngày
▶
  • 1.1
    OKR Review & AlignmentHead of Product + Engineering review OKR hiện tại. Xác định gaps và adjust nếu cần. Output: OKR doc được approve.CPO + HoE
  • 1.2
    Discovery Input CollectionThu thập: user feedback, support tickets, data analytics, competitor analysis. Mỗi PM present top 5 insights cho team.PMMixpanel / Hotjar
  • 1.3
    Initiative Scoring & PrioritizationScore theo ICE (Impact × Confidence ÷ Effort). Loại bỏ những thứ <20 điểm trừ khi có strategic reason đặc biệt.PM + EM
  • 1.4
    Capacity PlanningEM tính toán actual velocity từ 3 sprint gần nhất. Trừ 20% cho tech debt, bug, và unexpected. Commit roadmap theo capacity thực.Engineering ManagerMANDATORY
  • 1.5
    Roadmap Publish & CommunicationRoadmap được publish lên Confluence/Notion. Stakeholder brief qua async video (Loom). All-hands Q&A trong vòng 1 tuần.CPO
02
Feature Specification (PRD)
Mỗi initiative lớn · 3–5 ngày
▶
  • 2.1
    Problem StatementPM viết rõ: "Ai đang gặp vấn đề gì? Tại sao bây giờ? Metric hiện tại là bao nhiêu? Mục tiêu là bao nhiêu?"
  • 2.2
    User Stories + Acceptance CriteriaFormat: "As a [role], I want [action] so that [outcome]." Mỗi story phải có AC dạng Given/When/Then.REQUIRED
  • 2.3
    Design Mockup ReviewDesigner present wireframe/prototype. PM + Tech Lead + 1 engineer review. Feedback trong vòng 48h qua comment trên Figma.
  • 2.4
    Tech Design Document (TDD)Tech Lead viết TDD gồm: system design, API contract, DB schema changes, risk & mitigation. Review bởi senior engineers.Tech Lead
  • 2.5
    Definition of Ready CheckTicket chỉ được vào sprint khi: có AC, estimate, không có dependency open, design approved, TDD reviewed.GATE
03
Sprint Planning & Backlog Grooming
Mỗi 2 tuần · <2h
▶
  • 3.1
    Backlog Grooming (mid-sprint)PM + EM groom backlog trước sprint planning 3 ngày. Clarify, estimate, tách ticket quá lớn (>8 points → tách).Mid-sprint day 8
  • 3.2

  • 3.3

03
Code Review Process (Pull Request)
SLA: 1 business day first review
▶
  • 3.1
    PR Size LimitTối đa 400 lines changed (trừ auto-generated). PR lớn hơn phải được tách hoặc có lý do document rõ ràng.HARD RULE
  • 3.2
    PR Description TemplateBắt buộc: Link ticket, What changed & Why, How to test locally, Screenshot/Video nếu có UI change, Checklist (tests pass, no lint errors).
  • 3.3
    Reviewer AssignmentMinimum 1 approver (hoặc 2 nếu thay đổi core logic, security, hoặc DB schema). Tech Lead là required reviewer cho breaking changes.1 ngày làm việc
  • 3.4
    Review EtiquetteComments phải constructive và specific. Prefix: [MUST], [SUGGEST], [QUESTION], [NITPICK]. Tránh "this is wrong" — luôn giải thích tại sao và đề xuất cách khác.
  • 3.5
    Auto Checks (CI)Tất cả CI checks phải pass trước khi merge: Lint, Unit tests, Type check, Build, Security scan (Snyk/Dependabot).BLOCKING
  • 3.6
    Merge StrategySquash and merge cho feature branches (clean commit history trên develop/main). Rebase chỉ được dùng cho local cleanup.
04
Definition of Done (DoD)
Checklist bắt buộc trước khi close ticket
▶
  • Code đã qua review và được approve — tối thiểu 1 reviewer, PR merged
  • Unit tests đã được viết — coverage không giảm so với baseline
  • All CI checks pass — lint, test, build, type check
  • Acceptance Criteria được verify — QA hoặc dev tự test theo AC
  • Deploy lên Staging thành công — không có regression
  • Documentation cập nhật — API docs, README, hoặc Confluence nếu có thay đổi
  • Ticket status cập nhật — ticket moved to Done, linked với PR

QA & Testing Process
Chiến lược testing để đảm bảo chất lượng mà không làm chậm tốc độ phát triển.
🎯
Testing Pyramid: 70% Unit Tests · 20% Integration Tests · 10% E2E Tests. Đừng lộn ngược — E2E test nhiều = CI chậm, flaky, và tốn thời gian maintain.
UNIT COVERAGE
≥80%
Target tối thiểu cho critical paths
BUG ESCAPE RATE
<5%
Bugs được phát hiện sau khi release
TEST EXECUTION
<15m
Toàn bộ CI pipeline chạy xong
FLAKY TEST RATE
<2%
Test không ổn định phải được fix
01
Developer Testing (Shift Left)
Dev tự test trước khi tạo PR
▶
  • 1.1
    Unit TestingDev viết unit test cùng lúc với code (TDD hoặc ngay sau). Jest cho JS/TS. Mock external dependencies. Test edge cases, không chỉ happy path.
  • 1.2
    Integration TestingTest API endpoints với database thật (test DB). Supertest cho Express/Next.js. Verify contracts giữa services.
  • 1.3
    Local Smoke TestTrước khi tạo PR, dev phải tự test theo AC. Screenshot hoặc video evidence đính kèm vào PR.REQUIRED
02
QA Team Testing
Sau khi deploy lên Staging
▶
  • 2.1
    Test Case DesignQA viết test cases từ AC trong ticket. Format: Given/When/Then. Mỗi feature phải có test case trước khi dev bắt đầu code (BDD approach).BDD
  • 2.2
    Functional TestingVerify tất cả AC đều pass. Test cross-browser nếu có UI changes. Test responsive trên ít nhất mobile + desktop.
  • 2.3
    Regression TestingChạy regression suite cho các flows liên quan đến feature mới. Ưu tiên automated regression (Playwright/Cypress).Playwright
  • 2.4
    Bug ReportingBug report phải có: steps to reproduce, expected vs actual, environment, severity level, screenshot/video. Không accept bug report thiếu thông tin.REQUIRED FORMAT
  • 2.5
    Sign-offQA sign-off bằng cách comment "QA Approved" trên ticket. Không có QA sign-off → không được release.GATE
03
Performance & Security Testing
Trước mỗi major release
▶
  • 3.1
    Load Testingk6 hoặc Artillery test với expected peak load × 2. API phải handle p99 < 500ms dưới normal load.k6p99 < 500ms
  • 3.2
    Security ScanOWASP ZAP cho API security. Snyk cho dependency vulnerabilities. No high/critical CVE được phép vào production.Snyk + OWASP ZAP

03
Rollback Strategy
Phải thực hiện được trong <10 phút
▶
  • R1
    Feature FlagsMọi feature lớn phải được wrap trong feature flag. Kill switch để disable ngay khi có vấn đề mà không cần deploy.LaunchDarkly / UnleashBest Practice
  • R2
    Container RollbackDocker images được tag theo git SHA. Rollback = deploy lại image trước đó. Target: <5 phút.
  • R3
    Database RollbackNếu migration gây vấn đề, chạy down migration. Nếu không được, restore từ snapshot (phải có snapshot <1h trước deploy).RTO < 30 phút
  • R4
    Post-Deploy MonitoringMonitor 30 phút đầu sau deploy: error rate, latency, CPU/Memory. Alert nếu error rate tăng >50% so với baseline.

Incident & On-call Management
Phản ứng nhanh, có cấu trúc, học từ sự cố để không lặp lại.
Severity Levels & SLA
SEV-1
Production Down / Data LossToàn bộ service không hoạt động hoặc có nguy cơ mất dữ liệu. SLA: Acknowledge ≤15 phút, Resolve ≤4h. Wake up On-call ngay lập tức.
SEV-2
Major Feature BrokenCore feature không hoạt động cho nhiều user nhưng service vẫn sống. SLA: Acknowledge ≤30 phút, Resolve ≤8h trong giờ làm việc.
SEV-3
Minor DegradationFeature phụ bị ảnh hưởng hoặc performance giảm. SLA: Acknowledge ≤4h, Resolve trong sprint hiện tại hoặc kế tiếp.
SEV-4
Low Priority / CosmeticLỗi nhỏ không ảnh hưởng chức năng. Thêm vào backlog và prioritize theo quy trình bình thường.
Incident Response Playbook
▶
5 bước xử lý Incident
Áp dụng từ SEV-1 và SEV-2
▶
  • 1
    DETECT & DECLAREAi phát hiện trước → tạo incident channel trên Slack (#incident-YYYYMMDD-[desc]). Tag On-call engineer và EM. Đừng cố tự fix âm thầm.WITHIN 15 MIN
  • 2
    ASSIGN ROLESIncident Commander (IC): điều phối, không code. Tech Lead: investigate. Comms Lead: update stakeholder mỗi 30 phút. Không để 1 người làm tất cả.
  • 3
    MITIGATE FIRST, FIX SECONDƯu tiên giảm impact trước (feature flag off, rollback, scale up). Root cause có thể fix sau. "Stop the bleeding" là mục tiêu đầu tiên.
  • 4
    COMMUNICATEUpdate status page. Notify users bị ảnh hưởng (nếu cần). Internal update mỗi 30 phút. Clear, factual, không blame.
  • 5
    POST-MORTEMBắt buộc với SEV-1 và SEV-2. Trong vòng 48h sau khi resolve. Format: Timeline, Root Cause, Impact, Action Items (có owner và deadline). Blameless culture.MANDATORY SEV-1/2
📌
On-call Rotation: Mỗi engineer tham gia on-call rotation sau 6 tháng làm việc. Schedule công bố trước 2 tuần. Compensation: ngày bù hoặc phụ cấp theo policy công ty. Không để on-call rotation là "ai rảnh thì làm".
